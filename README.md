# add-to-sub.sh

Скрипт для 3x-ui, который позволяет объединить нескольких клиентов в **одну ссылку подписки**, даже на версиях панели, где это официально запрещено через веб-интерфейс.

## Проблема

Начиная с **3x-ui v3.2.5** панель запрещает присваивать одинаковый `subId` (ID подписки) разным клиентам — при попытке сохранить клиента с уже занятым `subId` вы увидите ошибку:

```
Something went wrong (Duplicate subId: ...)
```

До этой версии одинаковый `subId` можно было ставить нескольким клиентам сознательно — это штатный способ **объединить несколько отдельных VLESS/VMess/Trojan-ссылок в одну подписку**. Актуальный пример использования:

- Один VPS-сервер входа (например, в России) с единственным inbound на 443 порту
- Несколько клиентов в этом inbound, различающихся по `email`
- Routing-правила Xray, направляющие трафик каждого клиента на свой outbound (разные зарубежные сервера)
- Пользователь получает **одну** ссылку подписки, а в клиентском приложении (Happ, v2rayNG, Streisand и т.п.) видит сразу несколько готовых соединений — по одному на каждую "страну"/сервер

С v3.2.5 собрать такую подписку штатными средствами панели стало невозможно. Тема поднималась в issues автора 3x-ui ([#4776](https://github.com/MHSanaei/3x-ui/issues/4776), [#4759](https://github.com/MHSanaei/3x-ui/issues/4759), [#4775](https://github.com/MHSanaei/3x-ui/issues/4775)), но на момент написания скрипта официального решения в панели нет.

## Как это работает технически

В 3x-ui данные о клиенте фактически дублируются в двух местах:

1. **Таблица `clients`** в SQLite (`x-ui.db`) — вспомогательная, используется для статистики трафика и для проверки уникальности `subId`/`email` при сохранении через панель.
2. **JSON внутри колонки `settings` таблицы `inbounds`** (`{"clients": [...]}`) — это настоящий источник данных, который использует и Xray, и генератор подписки.

Проверка уникальности `subId`, введённая в v3.2.5, реализована только в коде записи через панель (`internal/web/service/client_crud.go`) и не является ограничением на уровне схемы БД — колонка `sub_id` в таблице `clients` имеет обычный индекс, а не `UNIQUE`-constraint. Это подтверждено чтением исходников 3x-ui.

Скрипт правит **оба места напрямую через `sqlite3`**, в обход панели:
- меняет `subId` нужного клиента внутри JSON в `inbounds.settings`;
- синхронно обновляет `sub_id` в таблице `clients` для консистентности данных, которые показывает панель.

После этого генератор подписки (`/sub/<subId>`) штатно собирает в одну подписку всех клиентов с этим `subId` — ровно так, как это работало до v3.2.5.

## Требования

- 3x-ui **v3.2.5 или новее** (на более старых версиях одинаковый `subId` можно поставить прямо в панели, без скрипта — он в этом случае откажется работать и подскажет об этом)
- root-доступ на сервере
- `sqlite3` и `jq` — при отсутствии скрипт попробует установить их через `apt`
- Путь к бинарнику 3x-ui: `/usr/local/x-ui/x-ui` (стандартный путь при установке через официальный `install.sh`)
- Путь к БД: `/etc/x-ui/x-ui.db` (стандартный для не-Docker установки; при необходимости меняется в начале скрипта)

## Установка

```bash
git clone https://github.com/diman-haritonov/add-to-sub.git
cd add-to-sub
chmod +x add-to-sub.sh
```

## Использование

```bash
sudo ./add-to-sub.sh <email_клиента> <target_sub_id>
```

Пример — объединяем трёх уже созданных и протестированных клиентов (`DE`, `NL`, `AMSTERDAM`) в одну подписку `ivan123`:

```bash
sudo ./add-to-sub.sh DE ivan123
sudo ./add-to-sub.sh NL ivan123
sudo ./add-to-sub.sh AMSTERDAM ivan123
```

Скрипт сам:
1. проверит версию 3x-ui и наличие нужных инструментов;
2. сделает бэкап `x-ui.db` перед правкой (`/etc/x-ui/backups/`);
3. остановит `x-ui`, внесёт правку, запустит обратно;
4. покажет итоговый список клиентов с этим `subId` для проверки.

Готовую ссылку подписки для пользователя берёте как обычно из панели (Клиенты → нужный клиент → Ссылки → Sub) — после склейки она будет отдавать все объединённые конфигурации разом.

## Важное предупреждение

**Создавайте и полностью тестируйте всех клиентов ДО склейки подписки.** После того как клиент получил общий `subId`, редактирование этого клиента через веб-панель (Клиенты → Изменить → Сохранить) будет завершаться ошибкой `Duplicate subId`, даже если вы меняете не сам `subId`, а, например, лимит трафика или срок действия.

Дальнейшие правки таких клиентов (трафик, срок, UUID и т.д.) нужно делать так же напрямую через `sqlite3`/`jq`, либо повторным запуском этого скрипта.

## Рекомендуемый порядок действий

1. Создайте один inbound (например, VLESS+Reality на 443)
2. Создайте в нём нужных клиентов с говорящими email (`DE`, `NL`, `AMSTERDAM` и т.п.)
3. Настройте routing rules Xray — матчинг по полю `user` (это и есть email клиента) на нужные outbound'ы
4. Протестируйте **каждого** клиента по отдельности — подключение, реальный внешний IP на выходе
5. И только после этого запускайте `add-to-sub.sh` для склейки в одну подписку

## Дисклеймер

Скрипт напрямую редактирует внутреннюю БД 3x-ui в обход штатной валидации панели. Автор не несёт ответственности за возможные проблемы — обязательно проверяйте бэкапы (`/etc/x-ui/backups/`) и тестируйте на некритичном окружении перед использованием в проде. Логика опирается на структуру БД версии 3x-ui на момент написания скрипта — при значительных изменениях схемы БД в будущих релизах 3x-ui может потребоваться доработка.
